Using jQuery Isn't More Minimal

Published 09. January 2021

Let's look at some JavaScript code:

document.getElementById("#foo");

and

$("#foo");

Which one would you consider to be the most minimalistic? The first example using vanilla JS or the jQuery example? I imagine that a lot of people would say the second example, because there's less on the screen, but I disagree, because there's more going on in the background. The first example using vanilla JS, is 32 bytes (0.032 KB). The jQuery example in and of itself is only 10 bytes. But that's if you look away from the jQuery module which by itself is (89 KB), with AJAX removed it's (72KB) (version 3.5.1). In other words, you just removed 89 KB from your site which can make a big difference depending on someones internet speed. Not to mention that jQuery is a layer of abstraction, so the vanilla JS code is actually more performant as well.

Examples

I've seen several examples where jQuery is being used simply for the use of selecting elements. One of my friends asked me why I won't just import jQuery instead of typing out all of that vanilla code each time. Well it's lighter and just as easy. Not to mention that "having to type out of all that vanilla code" is a bad excuse for choosing jQuery when you could just make functions for the few things you need:

function get(x)      {return document.querySelector(x);}
function getId(x)    {return document.getElementById(x);}
function getClass(x) {return document.getElementsByClassName(x);}
function getTag(x)   {return document.getElementsByTagName(x);}

You can copy-paste this between projects or create tiny modules for functionality like this. So if you're only using jQuery for getting elements, then the code above just saved you 89.2 KB, almost the entire jQuery module.

For something like AJAX requests, the use of jQuery becomes easier to understand as it can be quite tedious typing out a request in vanilla JS, especially if you're trying to reach a specific amount of browser compatibility. This is a good example of where jQuery shows its reasonable side:

$.ajax({
  type: 'POST',
  url: '/my/url',
  data: data
});

And yet, let's say you wanted to support IE8+ but send a POST request:

function requestPOST(url, data) {
  var request = new XMLHttpRequest();
  request.open('POST', url, true);
  request.setRequestHeader(
    'Content-Type', 
    'application/x-www-form-urlencoded; charset=UTF-8'
  );
  request.send(data);
}

That looks like a lot more to type out, but you only have to type out that function once.

It might look a bit intimidating to not use jQuery, but there are sites to convert code for you. I got the example above from YouMightNotNeedjQuery which shows how to write code with the same functionality as jQuery, and with different levels of compatibility going from IE8+ to IE10+.

I've only given specific uses so far, as in, "you could just do this if you're doing this one thing." Fair enough, let's create something bigger:

$("#wrapper").addClass("fade");
$(".fade").fadeIn();

$("container").each(function(i, el){
  el.hide();
});

$("#content").show();

This adds a fade class to a wrapper element, which then fades in. Then each element which has the tag, class or id container is hidden, and then the content is shown. Only 6 lines of code, and only 131 bytes.

Here's the vanilla JS equivalent (with IE8+ compatibility).

Functions:

function get(x)      {return document.querySelector(x);}
function getId(x)    {return document.getElementById(x);}
function getClass(x) {return document.getElementsByClassName(x);}

function addClass(el, className) {
  if (el.classList) {
    el.classList.add(className);
  } else {
    var current = el.className;
    var found = false;
    var all = current.split(' ');

    var i = 0;
    while (i < all.length && !found) {
      found = (all[i] === className);
      i += 1;
    }
    
    if (!found) {
      if (current === '') {
        el.className = className;
      } else {
        el.className += ' ' + className;
      }
    }
  }
}

function fadeIn(el) {
  var opacity = 0;

  el.style.opacity = 0;
  el.style.filter = '';

  var last = +new Date();

  var tick = function() {
    opacity += (new Date() - last) / 400;
    el.style.opacity = opacity;
    el.style.filter = 'alpha(opacity=' + (100 * opacity)|0 + ')';

    last = +new Date();

    if (opacity < 1) {
      (
        window.requestAnimationFrame && requestAnimationFrame(tick)
      ) || setTimeout(tick, 16);
    }
  };

  tick();
}

function forEachElement(elems, fn) {
  for (var i = 0; i < elems.length; i++) {
    fn(elems[i], i);
  }
}

Using the functions:

addClass(getTag("wrapper"), "fade");
fadeIn(getClass("fade");

forEachElement(get("container"), function(el, i) {
  el.style.display = 'none';
});

getTag("content").style.display = '';

Now that might look intimidating, but it really isn't. I copy pasted from YouMightNotNeedjQuery again, and changed the style a little. Not to mention the code actually calling the functions is only 6 lines of code as well, and yet we just saved 88.1 KB (which is still most of the jQuery module)! AND this is with IE8+ compatibility AND unminified. If you're only going for IE10+ support then you save even more.

Debloating the Web

Try taking a look at your own (or some other website) with PageSpeed Insights, and maybe even try to take a look at this site! There are several benefits to keeping things small.

  • Faster load times
  • More traffic since internet speed becomes less of a constraint
  • Better performance (if you write your code optimally)
  • Fewer dependencies
  • Easier maintenance
  • More control since you know what the code does exactly

There's another benefit I didn't list. Depending on the importance you think this topic has, this may or may not be of much importance to you. But the less data that has to be transferred and calculated on your site, the smaller your carbon footprint becomes. The fewer calculations the computer / servers have to do, a smaller amount of electrical power has to be generated by fossil fuels. You can check your website carbon footprint here: WebsiteCarbon. I didn't go for a low carbon footprint on purpose, but it was a positive side-effect of me optimizing my site for size and keeping everything completely minimal. And it was really easy to do as well!

Stack Overflow as a recommendation for bad practice

Stack Overflow for web-development with JS can be frustrating when you're trying to go against the wave of what I'd personally consider poor practice. There has been several occasions where I've tried to find a solution to a problem and a jQuery answer is marked as the solution. There's also several times where I need a reminder of how to do something in a language, then I search it up, click the Stack Overflow link and then I'm met with jQuery code instead of the vanilla JS solution, which in many cases happens to be just as simple and sometimes (but rarely) might even be simpler!
